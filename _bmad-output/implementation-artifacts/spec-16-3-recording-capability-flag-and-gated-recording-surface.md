## Intent

**Problem:** Recording is macOS-desktop-only (system-audio floor = macOS 13.0), but `CapabilitiesVm` has no `recording` flag and there is no macOS-version detection anywhere in the codebase. Without the flag and its gated surfaces, recording would either be absent everywhere or appear as dead buttons on platforms that cannot record — violating the "absent, never a dead button" invariant (AD-27).

**Approach:** Add a `recording: bool` to `CapabilitiesVm`, computed in the shell as *desktop macOS ≥ 13.0 only* via a new runtime version probe. Gate every recording surface on that flag: a ⌘5 Recording sidebar entry + primary-view, a Settings → Recording section, and a `open-recording` command-palette action (which transitively gates the cheat sheet and native menu bar through the single Rust registry). Ship the empty Recording view shell — a centered card stack of setup placeholders — ready for later stories. No capture, no live state, no color token yet.

## Boundaries & Constraints

**Always:**
- The `recording` flag is `true` **only** on desktop macOS ≥ 13.0. iOS (`target_os = "ios"`) and every non-macOS desktop → `false`. The app-wide `minimumSystemVersion` stays 11.0 (AD-35).
- Version detection lives in the **shell** crate (the platform adapter layer), keeping `keeper-core` free of `cfg(target_os)` / OS-version logic (AD-26). Any detection failure defaults the flag to `false` (safe-hide).
- `keeper-core` stays free of `unsafe`; the probe uses a dependency-free, `unsafe`-free mechanism (`sw_vers -productVersion`), not an Apple FFI binding.
- The frontend consumes the flag from `capabilitiesStore` only; it never sniffs `navigator.userAgent` or build flags (AD-35 — guarded by the existing `no-user-agent-gating.test.ts`).
- Surface gating is **additive/hiding** (the established `bridgeSidecar`/`nativeMenuBar` pattern): read `capabilities.recording`, render `{recording && <Surface/>}`. Never a disabled/dead affordance.
- The command-palette registry stays the single Rust source of truth (UX-DR15): the recording action is one entry in `palette_actions()`, gated by a context flag exactly like `requires_open_chat`, so palette + cheat sheet + menu bar can never drift.
- ts-rs `.ts` bindings are regenerated by `cargo nextest run` — never hand-edited.

**Block If:**
- The Recording view shell requires a design decision beyond an empty placeholder card stack (i.e. any real Source/Audio/Webcam/Destination/Segmenting/Advanced control) — that is later-story scope; do not invent it.

**Never:**
- No `recording-red` (#E5322D) color token, no `recording-dot`, no active-recording state, no elapsed line — all deferred to 16.6 (the empty shell has no live capture to indicate).
- No actual capture, sidecar spawn, RPC, or permission pre-flight (16.4/16.5/16.6).
- No recording surface on the phone tier (`PhoneShell` has no primary-view concept; the flag is already `false` there).
- No new `Platform`/`Recorder` port method — this story is capability + UI only; do not wire `Recorder::is_available()` into the flag (the flag is a pure OS-version gate).
- No `os_info`/objc2/AVFoundation dependency for version detection.

## I/O & Edge-Case Matrix

| Scenario | Input / State | Expected Output / Behavior | Error Handling |
|----------|--------------|---------------------------|----------------|
| macOS 13+ desktop | `sw_vers -productVersion` → `"14.5"` | `recording_supported()` → `true`; `CapabilitiesVm.recording = true`; all recording surfaces render | No error expected |
| macOS 12 desktop | `"12.7.1"` | `recording_supported()` → `false`; surfaces absent | No error expected |
| Legacy version string | `"10.16"` | parse major `10` → `false` | No error expected |
| iOS / non-macOS desktop | `target_os != "macos"` | compile-time `false` (probe not called) | No error expected |
| Probe fails | `sw_vers` missing / non-numeric / empty output | `parse_macos_major` → `None` → flag `false` (safe-hide) | Swallow, default false; no panic |
| Palette query, flag off | `recording_supported() == false` | `open-recording` filtered out of palette results, cheat sheet, and native menu | No error expected |

## Design Notes

**Version gate placement.** No macOS-version detection exists in the repo (no `os_info`, no `Platform` accessor). `capabilities()` is already the shell's platform-adapter site using `cfg!(desktop)`; the new probe belongs beside it. `sw_vers -productVersion` is the dependency-free, `unsafe`-free choice (objc2/`ProcessInfo` would need an FFI audit-inventory entry for no benefit here). Keep `parse_macos_major` pure so the version logic is unit-tested without a Mac; the subprocess wrapper is the only untested seam. Memoize the result in a `static OnceLock<bool>` — `palette_query` (per keystroke) and the menu build both re-query the flag, so `sw_vers` must not respawn each time.

```rust
pub fn parse_macos_major(v: &str) -> Option<u32> {
    v.trim().split('.').next()?.parse::<u32>().ok()
}
#[cfg(target_os = "macos")]
pub fn recording_supported() -> bool { macos_product_version().is_some_and(|m| m >= 13) }
#[cfg(not(target_os = "macos"))]
pub fn recording_supported() -> bool { false }
```

**Registry gating mirrors `requires_open_chat`.** The palette registry already filters context-sensitive actions with a bool threaded from the shell (`open_chat`). Adding `requires_recording` + a `recording: bool` param to `query`/`query_actions`/`registry_sections` reuses that exact seam, so the single Rust registry keeps palette, cheat sheet, and native menu perfectly consistent (UX-DR15) without any per-platform logic leaking into core — the shell computes the capability and passes it down.

**Layout.** Unlike `BridgesPane`'s full-bleed body, the Recording shell centers its content at `content-max-width` (`mx-auto w-full max-w-[720px]`, the `conversation-pane` realization) per UX-DR29, while reusing the same outer `<section>`/`<header>`/`<ScrollArea>` chrome for visual consistency with the other primary views.
