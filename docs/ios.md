# iOS build setup

keeper is a macOS-first Tauri 2 app that also targets iOS from the same codebase.
This document covers the local prerequisites, signing without secrets, and the
generate/regenerate loop for the Apple project under
`src-tauri/crates/keeper/gen/apple/`. Booting the app in the Simulator is out of
scope here — that arrives with the desktop/mobile compile seam in story 12.2.

## Prerequisites

- **Xcode** (26.x used to generate the current project) with the iOS SDK and
  `xcodebuild` on `PATH`. Run `xcodebuild -runFirstLaunch` once after install.
- **CocoaPods** and **XcodeGen** via Homebrew:

  ```sh
  brew install cocoapods xcodegen
  ```

- **iOS Rust targets** (pinned in `rust-toolchain.toml`, so `rustup` installs them
  on first use in-tree; install explicitly if needed):

  ```sh
  rustup target add aarch64-apple-ios aarch64-apple-ios-sim
  ```

- **bun** for all JS tooling (never npm/yarn/pnpm) and **tauri-cli 2.x**
  (`bun run tauri --version`).

## Signing without secrets

No Apple Team ID or signing identity is ever committed. The generated project uses
the generic `iPhone Developer` identity and no `DEVELOPMENT_TEAM`. For local device
builds, export your team id in your shell — never write it into any tracked file:

```sh
export APPLE_DEVELOPMENT_TEAM=XXXXXXXXXX   # your 10-char Apple Team ID
```

Simulator builds do not require a team. Keep team ids, provisioning profiles, and
`.p12` identities out of the repo entirely.

## Project generation and the regeneration loop

The Apple project lives at `src-tauri/crates/keeper/gen/apple/`. It is generated by
Tauri and is committed **except** ephemeral output (`build/`, `Externals/`, `Pods/`,
`.xcode/`, `xcuserdata/`), which `src-tauri/.gitignore` excludes (AD-32).

`project.yml` is the **source of truth**. The `.xcodeproj` is a derived artifact that
**XcodeGen** regenerates from `project.yml` — never hand-edit `project.pbxproj`. Make
persistent changes only in `project.yml`, `keeper_iOS/Info.plist`, and the
`Sources/`/`keeper_iOS/` files, then regenerate.

Initial generation (run from the repo root, using the shared config the other
scripts use) is a **one-time** bootstrap — the `gen/apple/` tree is already committed,
so you do **not** re-run `init` in normal work (re-running it overwrites committed
`project.yml`/`Podfile` edits). Use it only to bootstrap a brand-new Apple project:

```sh
bun run tauri ios init --config src-tauri/crates/keeper/tauri.conf.json
```

Regenerate the `.xcodeproj` from `project.yml` after any edit:

```sh
cd src-tauri/crates/keeper/gen/apple && xcodegen generate
```

Regeneration is lossless: a fresh checkout plus `xcodegen generate` reproduces the
committed `project.pbxproj` with no drift. This byte-for-byte guarantee is
XcodeGen-version-sensitive — the committed project was generated with **XcodeGen
2.45.4**; a different major version may reshape `project.pbxproj` and require a
re-commit of the regenerated project.

## Deployment target

Minimum iOS is **16.0**, set in `project.yml` (`options.deploymentTarget.iOS`) and the
`Podfile`, and propagated into the project by `xcodegen generate`. The bundle
identifier `dev.tgorka.keeper` is shared with the macOS build.

## Verifying the core compiles for iOS

The Tauri-free core compiles for iOS independently of the shell's compile seam:

```sh
cd src-tauri && cargo check --target aarch64-apple-ios -p keeper-core
```

Building and booting the full shell in the Simulator (`tauri ios dev`) depends on the
desktop-only surfaces being cfg-gated out — that is story 12.2's exit criterion.
